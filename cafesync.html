Backend Stack

FastAPI + Uvicorn — Python ASGI framework with async request handling. Dependency Injection pattern enforces authorization at the router level, not the controller. SQLAlchemy ORM abstracts all database interaction with parameterized queries.

Frontend Stack

Jinja2 Server-Side Rendering — chosen deliberately over a SPA framework. Eliminates the CORS attack surface entirely and ensures no authentication token is ever stored in localStorage. CSP nonce validation blocks injected scripts regardless of origin.

Design Philosophy

Every security control is independently effective. UI hiding is a UX convenience — the server enforces all permissions via signed sessions and dependency injection, even against a raw HTTP client.

Perimeter Hardening

Rate Limiting · CSRF · Content Security Policy · Security Headers

Asynchronous Rate Limiter

SlowAPI middleware enforces per-IP request budgets. Custom client-IP extraction respects Fly-Client-IP reverse-proxy headers. Localhost requests receive unique per-request keys to avoid false positives in development.

CSRF Token Architecture

Per-session 256-bit token via secrets.token_urlsafe(32). Validated using secrets.compare_digest — constant-time comparison to defeat timing side-channel attacks. Accepted via form field or X-CSRF-Token header.

GET / HEAD / OPTIONS → EXEMPT (RFC 7231)
POST / PUT / DELETE → TOKEN REQUIRED

Credential & Session Security

bcrypt Hardening · Timing-Attack Mitigation · Session Lifecycle

bcrypt Parameterization

All passwords are hashed with bcrypt at cost factor 12. An explicit 72-byte truncation guard prevents the bcrypt password-length vulnerability — passwords exceeding 72 bytes are rejected at registration, not silently truncated.

Unknown-username login paths execute a dummy bcrypt hash against a static sentinel value — ensuring the response latency is indistinguishable from a real hash verification. This closes the timing side-channel that would otherwise confirm username existence.

rounds = 12
max_len = 72 bytes
dummy hash on miss
Session Lifecycle Management

Sessions are signed with itsdangerous.URLSafeTimedSerializer using a 256-bit SESSION_SECRET environment variable. An expires_at timestamp is stored inside the session and re-validated on every request by the auth-gate middleware — a tampered expiry is detected because the signature would not verify.

Production cookie flags:
  https_only = True
  same_site = "strict"
  signed = itsdangerous HMAC
  TTL = 8 h / 30 d (Remember Me)
Technical Justification

The dummy-hash approach costs one bcrypt operation per failed login for an unknown username — an intentional performance cost that buys a meaningful security guarantee. Without it, an attacker performs user enumeration in O(n) network requests by measuring response time differentials, reducing the search space before a brute-force campaign.

Data Layer Integrity

SQL Injection Prevention · Error Disclosure · Schema Boundary Validation

Parameterized ORM Queries

SQLAlchemy ORM exclusively — zero raw SQL strings exist anywhere in the codebase. All user-supplied values pass through SQLAlchemy's bound-parameter mechanism, preventing both classic and second-order SQL injection regardless of the database engine.

db.query(User)
  .filter(User.username == username)
  .first()

Portable guarantee — survives migration to SQL Server.

Error Disclosure Prevention

A global exception handler in main.py intercepts all unhandled exceptions. When DEBUG_ERRORS ≠ 1 (production default), the response body is always:

{"detail": "Internal server error"}

Stack traces, module paths, and database error messages are never transmitted to the client, preventing reconnaissance via error-driven enumeration.

Schema Validation at System Boundary

Pydantic v2 models enforce type, length, and format constraints on every API input before it reaches business logic. Malformed payloads (oversized strings, wrong types, missing fields) are rejected at the framework deserialization layer with a structured 422 response.

Combined with ORM parameterization, this creates a dual-layer defense: Pydantic rejects structurally invalid input; SQLAlchemy neutralizes any surviving SQL metacharacters.

Defense-in-Depth Principle

Even if a route handler contained a logic bug, the ORM layer prevents injection. Even if injection succeeded, the generic error handler prevents stack-trace disclosure. No single vulnerability results in full compromise — each layer degrades the attacker's information gain.

Multi-Vector Authentication BONUS

TOTP · Backup Codes · WebAuthn / FIDO2 Passkeys

TOTP — RFC 6238

Implemented via pyotp.TOTP with a 30-second window and ±1 step tolerance for clock skew. Setup flow: server generates a TOTP secret → renders a QR code PNG → user scans with Google Authenticator. Challenge is gated in a separate pending_login session state with a 5-minute TTL. Rate-limited at 5 attempts/min/IP.

30s WINDOW · RFC 6238
Backup Codes

10 single-use recovery codes are generated per user on 2FA setup. Each code is hashed with bcrypt before storage — plaintext is shown exactly once at generation time. On redemption, the used flag and used_at timestamp are recorded atomically, enforcing single-use semantics. Regeneration invalidates all previous codes immediately.

10 CODES · BCRYPT-HASHED · SINGLE-USE
WebAuthn / FIDO2 Passkeys

Passwordless login via hardware security keys or device biometrics (Touch ID, Windows Hello). The credential's public key is stored as a LargeBinary field. On every authentication, the presented sign_count must exceed the stored value — a cloned credential presents a stale counter and is rejected, preventing replay attacks. Origin binding to RP_ID blocks cross-domain credential presentation.

Authorization is enforced via FastAPI's Dependency Injection system. Each protected router declares a dependency (require_admin, require_fulfillment, require_dashboard) that extracts the role from the signed session and raises HTTP 403 before any business logic executes. VPS / Self-Managed Host Rationale

POS compliance requires full auditability of the request path. Managed PaaS environments (Vercel, Railway) abstract SSL termination and packet routing — operators lose visibility into the network layer. A self-managed Linux host (or Docker on a VPS) preserves that visibility and enables direct hardening:

UFW → deny all inbound except 80/443
Fail2Ban → auto-ban on repeated 401/429
SSH → key-only (PasswordAuthentication no)
Container → non-root user, read-only FS
Brute-force: Rate limit (5/min) + Fail2Ban IP ban  ·  MITM: HSTS preload (max-age=31536000)  ·  Session theft: Secure + HttpOnly + SameSite=strict  ·  XSS: CSP nonce per-request  ·  CSRF: 256-bit session-bound token  ·  SQLi: ORM parameterization  ·  Info leak: Generic 500 handler

Philosophy — Why No Single Layer is Trusted

The rate limiter, CSRF guard, RBAC dependency, bcrypt hasher, ORM parameterizer, and generic error handler are each independently sufficient for their respective threat class. An attacker who bypasses the rate limiter still faces a valid CSRF token requirement. An attacker who forges a CSRF token still faces server-side role enforcement. An attacker who escalates role still cannot read another user's bcrypt hash. Compromise of one layer does not cascade.

Technical FAQ

5 Advanced Questions — Professor Review Preparation

Q1 How does your rate limiting prevent distributed brute-force attacks, and what is its acknowledged limitation?
SlowAPI enforces per-IP limits (5 req/min on /login) using an in-memory counter keyed by client IP, extracted from the Fly-Client-IP proxy header. Acknowledged limitation: the counter resets on process restart and is not shared across replicas — a distributed attacker splitting requests across IP addresses bypasses per-IP thresholds. Production mitigation: Fail2Ban on the host detects patterns across the 401 log stream and imposes IP-level bans at the network layer, independent of the application counter.
Q2 Explain your CSRF protection mechanism and why GET requests are exempt.
Every state-changing request (POST/PUT/DELETE) must carry a session-bound 256-bit token (secrets.token_urlsafe(32)) either as a form field or X-CSRF-Token header. Validation uses secrets.compare_digest — constant-time comparison to defeat timing side-channel attacks. GET/HEAD/OPTIONS are exempt per RFC 7231: safe methods must be idempotent and carry no state-changing semantics, so exempting them avoids breaking browser prefetch and navigation without weakening protection where it is actually needed.
Q3 How does WebAuthn prevent credential replay and cloning attacks?
The passkeys table stores a sign_count integer alongside each credential's public key. On every successful authentication, the authenticator increments its internal counter and includes the new value in the signed assertion. The server rejects any assertion where the presented sign_count ≤ the stored value — a cloned credential will present a stale counter and is rejected immediately. Additionally, origin binding to RP_ID ensures the credential is only accepted from the exact domain it was registered against, blocking cross-site credential presentation.
Q4 How do you prevent a compromised BARISTA account from escalating privileges to ADMIN?
Authorization is enforced via FastAPI Dependency Injection. The require_admin dependency on /users and /telemetry routes reads the user's role from the signed, server-verified session cookie and raises HTTP 403 before any business logic executes — even against a raw crafted HTTP request. The front-end hiding of admin controls is purely a UX layer. Additionally, the role-change endpoint enforces that a user cannot modify their own role, closing self-escalation via a race condition or social engineering.
Q5 What is your SQL injection strategy, and does SQLite change the threat model?
The application uses SQLAlchemy ORM exclusively — zero raw SQL strings exist anywhere in the codebase; all user-supplied values pass through SQLAlchemy's bound-parameter mechanism. SQLite does alter the threat model marginally: unlike PostgreSQL, most SQLite drivers do not support stacked queries via ;, reducing potential blast radius. However, relying on that would be defense-by-obscurity — the ORM parameterization is the authoritative, engine-agnostic control. This ensures the security guarantee survives a migration to SQL Server (documented in README.md as the planned production database).
